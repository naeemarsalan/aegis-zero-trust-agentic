# Phase-A delegated-agent journey — work log + issues resolved (2026-06-20 → 06-21)

Chronological record of what was done to get a credential-less OpenShell sandbox agent reading
delegated + writing JIT-gated, every blocker hit, and how each was resolved. Companion to
[ADR-0018](../adr/0018-openshell-native-svid-grant-key-binding.md),
[ADR-0013](../adr/0013-kagenti-identity-plane-adoption.md),
[ADR-0017](../adr/0017-provider-spiffe-setns-selinux-confinement.md), and the
[Phase-A plan](../plans/phase-A-openshell-native-loops.md) /
[enrollment decision-map](../plans/option2-kagenti-enrollment-decision-map.md).

## Goal
A credential-less agent in a gateway-launched OpenShell sandbox: **reads** with a delegated identity,
**writes** only with human-approved, just-in-time, short-lived elevation. The agent holds **only its
SPIFFE SVID** — no stored broadly-scoped credential. This single loop is the Phase-A gate; B/C are blocked on it.

## Outcome (updated 2026-06-21 after the E2E workflow)

CAVEAT: this ran in the brain-enabled **`e2e-harness` pod (agent-sandbox ns / ext-proc plane against the REAL pfSense tool)**, not the OpenShell-native sandbox — see issue #16.
- **`test-openshift-jit.sh` = 4/4** (OpenRouter credits restored).
- **OpenShell is OPERATIONAL (2026-06-21):** a gateway-launched **native** sandbox now boots and runs a **real LLM agent** (image swap to `agent-harness` + `ExecSandbox` drive + `PYTHONPATH=/app/src`; the gateway reserves the command env, issue #19). - **🎯 UNIFIED PHASE-A GOAL MET (2026-06-21, workflow `wxhf010s7`):** an **LLM agent inside a native OpenShell sandbox** completed the **real pfSense** journey via ext-proc — SVID `…/ns/openshell/sandbox/3d2e5114-…` (UUID plane) → read `search_firewall_rules` **200** (`grant_result=valid, credential_injected=true`) → write `create_firewall_rule_advanced` **403** `grant_scope_denied` → human-approve → write **200** (`jit_elevated=true`, real pfSense rule id 47 created). **Both identity planes coexist** on each pod: SA-shaped (Kagenti/echo-mcp) + UUID-shaped (ext-proc/real tools), 2 SVID entries/pod, no interference. ## Strategic pivot recorded mid-session
The delegated-read path moved from the **bespoke ext-proc/Vault-grant** design to **Kagenti AuthBridge**
(the chosen identity plane, ADR-0013). Consequence: the per-sandbox UUID SVID + the launcher's Vault
grant (Loops 1–2 below) are **off the delegated-read path** on the Kagenti model (which is
agent-attributed identity + our JIT, not on-behalf-of-the-user). Loop-1's SVID issuance stayed valid
(Kagenti's spiffe-helper needs the SVID); Loop-2's grant is now ext-proc-only (retired path).

### Major correction — KC OBO is NOT a dead-end (2026-06-21, workflow `wh25z0ez6`, read-only + isolated proof)
The long-held belief that real per-user Keycloak OBO is blocked by the RHBK #40328 NPE is **stale for this build**. On the live RHBK 26.6.3 there are **zero NPEs**; ext-proc's exchange hits a clean **v1 policy denial** (`error="not_allowed" reason="client not allowed to impersonate"`, HTTP 403) purely because the agentic `mcp-gateway` client has **fine-grained admin permissions disabled** (+ no impersonation policy). The features are already enabled (token-exchange:v1 + admin-fine-grained-authz:v1 + spiffe:v1). **Proven in the isolated `kagenti` realm (then fully reverted, agentic untouched):** enabling fine-grained perms + binding a client policy to the token-exchange + users-`impersonate` scope permissions makes ext-proc's exact naked-impersonation request (`requested_subject`, no `subject_token`) return **HTTP 200, `sub=user`** — **no code change, no feature-flag change.** So real per-user OBO is a per-client config away (additive/reversible on the agentic `mcp-gateway` client id `17452fe4-…`). Caveat: v1 token-exchange is deprecated (shelf life); the durable end-state is v2 `subject_token` (needs launcher token-carry). **Recommendation:** keep the static-token fallback for the PoC; schedule the v1-policy enablement (proven) as the near-term OBO milestone (off-hours + snapshot); plan the v2 migration as the durable end-state. Memory `project-keycloak-obo-constraint` updated.

## Environment gotchas learned (save future time)
- **`oc delete` of CLUSTER-SCOPED resources is HARD-DENIED** by the harness permission policy (clusterpolicy/clusterrole/clusterspiffeid). Namespaced deletes (sandbox, networkpolicy) work. This is why several rollback commands were "denied" — not the user declining. Cluster-scoped reaping must be done by the user.
- **Never whole-file `oc apply` `cluster-spiffe-ids.yaml`** — the harness CSID drifts from git (issue #4). Apply per-doc.
- **OVN-K DNS egress** needs `:53` AND `:5353` to `openshift-dns` (issue #6).
- **Webhook objectSelector ordering** — a label a later mutating webhook adds does NOT retroactively trigger a selector-gated earlier webhook (issue #8). Set selector labels at the resource's creation source.
- **Workflow subagents** have a more restricted Bash permission than the main agent (cluster-scoped deletes blocked for them too).
