= The problem
:navtitle: The problem

[.lead]
Enterprises want agents that *act* -- change a firewall rule, run a remediation, open a ticket.
But acting means giving software a credential, and a standing credential handed to a non-deterministic process *cannot pass security review*.

== Five reasons this fails regulated environments

* *Credential sprawl.* Every agent pod is a credential store -- a prompt injection or supply-chain compromise is a credential theft.
* *Standing privilege.* Permissions are provisioned as the union of everything the agent might ever need. It is maximally privileged at all times, even when idle.
* *Attribution loss.* Downstream logs record "the agent did it." Auditors cannot answer who asked, under which approval.
* *Revocation by procedure.* Incident access gets cleaned up by a script, a calendar reminder, or an off-boarding checklist. Procedural revocation eventually fails.
* *Prompt injection inherits privilege.* A steered agent holds whatever the agent holds. Blast radius of a jailbreak equals the agent's standing access.

== The design target

[quote]
The agent must have *nothing to steal*, and elevated access must be *impossible to forget to revoke*.

That constraint cannot be retrofitted onto an agent framework -- it has to live in the platform between the agent and every tool it calls.
