= Observability — OTel, Loki, AlertManager
:navtitle: Observability

[.lead]
The observability stack provides unified metrics, logs, and alerting for all Aegis platform components and AI agent workloads.
It:

* Collects OpenTelemetry traces and metrics from agentgateway, ext-proc-delegation, jit-approver, and agent pods
* Ships logs to an external Loki instance with **tool arguments SHA-256 hashed** — raw args never reach any log sink
* Exposes dashboards in an existing Grafana instance
* Feeds AlertManager rules to the EDA-AAP remediation loop (UC2 self-healing)

The platform does not deploy a new Grafana or Loki instance — both exist at `172.16.2.252` as shared homelab infrastructure and are parameterized via `LOKI_PUSH_URL` / `GRAFANA_URL` in the kustomize overlay.

== Audit invariant

Every log line emitted to Loki from this platform has tool arguments **replaced with their SHA-256 hash** before leaving the cluster.
This is enforced by an OTel Collector Transform processor in the pipeline:

[source,yaml]
----
# OTel Collector Transform processor
transform/hash_args:
  log_statements:
    - context: log
      statements:
        - set(attributes["tool_args"],
            Concat(["sha256:", SHA256(attributes["tool_args"])], ""))
----

Raw argument values are never written to any log sink.
This is verifiable by querying Loki and confirming all `tool_args` values begin with `sha256:`.

== EDA denial loop

When a Kyverno ext_authz decision or a pfsense-mcp response triggers a denial metric, the UWM AlertManager rule fires a webhook to the AAP EDA Event Stream on `hammer`.
This is the automated half of the self-healing loop — the EDA rulebook matches the alert and triggers a remediation job template which opens a Gitea PR for human review.

The alert → EDA → PR path is the automated trigger for UC2: the remediation itself cannot proceed without a human merge.

== Security posture

* **SPIFFE ID:** `spiffe://anaeem.na-launch.com/ns/agentic-observability/sa/otel-collector`
* **Loki credentials** (if Loki requires auth): injected via Vault Agent Injector (tmpfs); not in ConfigMaps
* **Fail-mode:** if OTel Collector is unavailable, platform components use buffered export with retry; if buffer exhausts, spans are dropped (observability is non-critical-path — fail-open for the data plane, but an alert fires)

*NetworkPolicy:* ingress on 4317/4318 from `mcp-gateway`, `agentic-mcp`, `agent-sandbox`, `keycloak`, `vault` namespaces; ingress on 9093 from `mcp-gateway` (AlertManager webhook); egress to Loki (`172.16.2.252:3100`) and Grafana (`172.16.2.252:3000`); deny all other ingress.

== Verify

[source,bash]
----
# 1. Check OTel Collector pod is Running
oc get pods -n agentic-observability -l app=otel-collector

# 2. Confirm logs reach Loki (count recent entries from mcp-gateway namespace)
curl -s "http://172.16.2.252:3100/loki/api/v1/query_range" \
  --data-urlencode 'query={namespace="mcp-gateway"}' \
  --data-urlencode 'start=1h' | jq '.data.result | length'

# 3. Verify tool_args are hashed in a sample log line (MUST NOT contain raw JSON)
curl -s "http://172.16.2.252:3100/loki/api/v1/query" \
  --data-urlencode 'query={app="agentgateway"} |= "tool_args"' \
  | jq '.data.result[0].values[-1][1]' | grep "sha256:"
----

== Maturity flags

* OpenTelemetry Collector (contrib distribution) is GA; deployed as a standalone Deployment rather than via the OTel Operator for simplicity
* Loki push endpoint port 3100 — confirm with the Loki admin before applying; the OTel Collector's `loki` exporter supports both the native Loki push endpoint and Grafana Agent port formats
* KEDA is already installed on `anaeem` and can scale agent pods based on queue-depth metrics from the OTel pipeline if needed
