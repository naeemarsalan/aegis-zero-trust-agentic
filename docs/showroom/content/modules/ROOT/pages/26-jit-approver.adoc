= jit-approver — JIT Approval Gate
:navtitle: jit-approver

[.lead]
`jit-approver` implements the human-in-the-loop Just-In-Time approval gate for privileged MCP tool calls and elevated Kubernetes access (UC2).
When an agent is denied access, it requests a scoped, time-boxed grant through `jit-approver`, which:

. Validates the requested scope against the configured ceiling
. Creates a pending approval record in a CNPG (PostgreSQL) WORM audit table
. Presents the request in the **Approval Console** (mint-gate UI) for a human reviewer
. Enforces **approver ≠ requester** at mint time — the requester cannot self-approve
. On approval, calls the Vault Kubernetes secrets engine to mint an ephemeral SA + Role + RoleBinding
. Mints a signed RS256 **session-capability JWT** for the Kyverno gate with scope-bound TTL (write = 5 min single-use, exec = 30 min)
. Writes both credentials to `secret/data/jit/<session>` in Vault KV
. Returns credentials to the agent via `GET /requests/\{id}/status` over SVID-mTLS
. Writes a tamper-evident outcome record (approved/denied, approver identity, scope, TTL) to the WORM audit log
. Emits an OTel span when the session closes; Vault lease expiry auto-revokes the SA+Role+RoleBinding

== Approval flow summary

[source]
----
Agent denied → POST /request {scope, session-id, justification}
  ↓ ceiling check → reject if over-ceiling
jit-approver creates pending record in CNPG WORM audit table
  ↓ Approval Console shows request to human reviewer
Human approves in console (approver ≠ requester enforced at mint)
  ↓ scope re-validated against ceiling
jit-approver calls Vault kubernetes engine → mint SA+Role+RoleBinding
jit-approver mints RS256 session-capability JWT
  (write scope: 5 min TTL, single-use jti; exec scope: 30 min TTL)
  ↓ writes {sa_token, session_jwt} to secret/data/jit/<session>
  ↓ WORM outcome record written (approver, scope, TTL, timestamp)
Agent polls GET /requests/\{id}/status → receives credentials
  ↓ session window expires
Vault lease TTL deletes SA+Role+RoleBinding (auto-revoke, structural)
----

== Session-capability JWT (ADR 0006)

When a session is issued, `jit-approver` mints an RS256 JWT:

[source,json]
----
{
  "iss": "https://jit-approver.mcp-gateway.svc.cluster.local:8080",
  "aud": ["kyverno-authz"],
  "sub": "<session_id>",
  "tool_scope": ["pfsense.block_ip"],
  "jti": "<single-use-id>",
  "exp": <now + TTL>
}
----

TTLs are operation-shaped and non-negotiable:

* **write** — 5 minutes, `jti` is consumed on first use (single-use, enforced via CNPG `consumed_jti` table)
* **exec** — 30 minutes

Vault lease expiry auto-revokes the backing SA+Role+RoleBinding regardless of the JWT TTL — there is no cron job or manual step required.

The agent presents this as `X-JIT-Session-JWT` on dangerous tool calls through the gateway.
Kyverno fetches the JWKS from `/jwks` and verifies the JWT stateless — no live HTTP callback to `jit-approver` per request.
A missing, invalid, expired, or already-consumed JWT → 403.

The JWT is the agent's **own scoped capability** — it does not carry any downstream service credential and does not violate the no-credential-passing invariant.

== Security posture

* **SPIFFE ID:** `spiffe://anaeem.na-launch.com/ns/mcp-gateway/sa/jit-approver`
* **Approver ≠ requester:** enforced at mint time in `mint_core`; the requester's identity is recorded on the request record and blocked from approving their own request
* **WORM audit:** all approval records are written to CNPG (`mint_core` schema) as insert-only rows; no update or delete path exists in the schema
* **Approval state:** durable in CNPG — survives pod restarts; if no approval arrives within the configured timeout, `jit-approver` returns DENY — fail-closed
* **RS256 signing key:** persisted in Vault (not ephemeral in-process memory) so JWTs issued before a restart remain verifiable after it
* **Auto-revoke:** Vault lease expiry deletes the SA+Role+RoleBinding structurally — no cron job required

*NetworkPolicy:* ingress on 8080 from `ext-proc-delegation` pod only; ingress on console Route port from OCP router; egress to Vault:8200, CNPG:5432, Alertmanager:9093; deny all other.

== Maturity flags

WARNING: `jit-approver` and the Approval Console are bespoke Python services.
Consider them PoC-grade pending external security review.

* Approval state is durable in CNPG — pod restarts do not lose pending approvals
* The RS256 signing key rotation procedure requires a brief JWKS overlap window to avoid rejecting in-flight JWTs
* `consumed_jti` enforcement (single-use write tokens) depends on CNPG being available; if CNPG is unreachable, write-scope mints are blocked fail-closed
