= Agent Sandbox — Native OpenShell Isolation
:navtitle: Agent Sandbox

[.lead]
The agent sandbox is the isolated execution environment for all AI agent workloads.
Today the platform uses **native OpenShell sandboxes** — each agent boots inside a confined pod managed by the OpenShell operator, carrying its own SPIFFE SVID, no SA-token automount, and a default-deny NetworkPolicy that restricts egress to the MCP gateway only.

This is the proven runtime: a credential-less LLM agent in a native OpenShell sandbox performs the full zero-trust journey — read-delegated, write-denied, human-approved, JIT-elevated write, capability-expired.

== SVID topology

Each OpenShell sandbox carries **two SVIDs** via two ClusterSPIFFEIDs:

[source]
----
openshell-sandbox-workloads  →  SA-shaped SVID
  spiffe://anaeem.na-launch.com/ns/openshell/sa/openshell-sandbox
  Used by: Kagenti AuthBridge → Keycloak federated-jwt → jit-gate → echo-mcp

openshell-sandbox-extproc    →  UUID-shaped SVID
  spiffe://anaeem.na-launch.com/ns/openshell/sandbox/<uuid>
  Used by: ext-proc-delegation → Vault consent grant → real tools (pfSense, k8s)
----

The agent's `svid_bearer` selects the UUID SVID for ext-proc via `SVID_REQUIRE_PATH_SUBSTR=/sandbox/`.

== Security posture

* **No standing credentials:** agents hold only their JWT-SVIDs; no API keys, no SA tokens, no pfSense credentials
* **Vault consent grant:** the sandbox launcher writes a per-sandbox grant at `secret/data/sandbox-grants/<uuid>` before the agent boots; ext-proc-delegation reads it at request time
* **Fail-mode:** if the SPIRE Workload-API is unavailable, SVID fetch fails and the agent cannot authenticate — no fallback to static credentials
* **setns fix (ADR-0017):** `CAP_SYS_CHROOT` is added via Kyverno mutation (`mutate-openshell-sandbox-syschroot`); sandbox remains container_t + MCS confined

*NetworkPolicy:* default-deny in `openshell`; explicit egress to `mcp-gateway.apps.anaeem.na-launch.com:443` only; no direct egress to the internet or to other namespaces.

== Verify

[source,bash]
----
# 1. Confirm an OpenShell sandbox is Running
oc get sandbox -n openshell

# 2. Confirm the UUID SVID is issued to the sandbox pod
oc exec -n openshell <sandbox-pod> -- \
  /opt/spire/bin/spire-agent api fetch jwt \
  -socketPath /run/spire/sockets/agent.sock -audience test | grep "sandbox/"

# 3. Confirm no SA-token volume is mounted on an agent pod
oc get pod -n openshell <pod> -o json \
  | jq '.spec.automountServiceAccountToken'
# Expected: false

# 4. Confirm the Vault consent grant was written for this sandbox
SANDBOX_UUID=$(oc get pod -n openshell <pod> -o jsonpath='{.metadata.annotations.openshell\.io/sandbox-id}')
oc exec -n vault vault-0 -- vault kv get secret/sandbox-grants/$SANDBOX_UUID
----

== Hardening roadmap — Kata and Confidential Containers

Native OpenShell (runc + MCS) is the current runtime.
The hardening roadmap adds two further isolation tiers when hardware is available:

[cols="1,2,1",options="header"]
|===
| Tier | Mechanism | Status

| Native OpenShell
| runc + MCS confined; no host-kernel sharing with other tenants
| **Live (current)**

| Kata Containers
| Lightweight KVM micro-VM per pod (`kata-qemu`); separate kernel
| Roadmap — requires `nested virt` or bare-metal; no identity or policy redesign needed

| Confidential Containers (CoCo)
| Kata peer-pods + hardware TEE attestation (Intel TDX / AMD SEV-SNP); encrypted VM memory
| Roadmap — requires TEE-capable hardware; see xref:32-confidential-containers.adoc[Confidential Containers]
|===

Moving from native OpenShell to Kata or CoCo requires only a `runtimeClassName` change and (for CoCo) Trustee KBS deployment — the SPIRE trust domain, Keycloak realm, Vault policies, and Kyverno rules are unchanged.

== Maturity flags

* Native OpenShell sandboxes are the proven PoC runtime; Kagenti + ext-proc dual-SVID topology is validated end-to-end
* `provider_spiffe` is enabled live (ADR-0017 setns fix applied)
* Kata `stable` channel is GA on OCP 4.20 but is a roadmap item for this platform — not the current runtime
