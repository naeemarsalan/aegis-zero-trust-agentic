= The solution
:navtitle: The solution

[.lead]
Aegis is a zero-trust identity fabric for agentic AI on OpenShift. Agents
authenticate with a *workload identity that cannot be exfiltrated as a
credential*; a gateway swaps that identity for user-scoped, short-lived
access *outside* the agent's reach; and privileged actions require a human
approval that materializes as a *self-destructing Kubernetes identity*.

== The shape of a tool call

Every MCP tool call from an agent crosses the same pipeline:

[mermaid,target=aegis-pipeline,format=svg]
....
flowchart LR
    classDef step fill:#1565c0,color:#fff
    classDef art fill:#2e7d32,color:#fff
    classDef gate fill:#f57f17,color:#000

    A[Agent pod\nKata-isolated\nSPIFFE SVID only]:::art --> G[agentgateway\nJWT authn]:::step
    G --> K{Kyverno ext_authz\nis THIS tool allowed?}:::gate
    K --> X[ext-proc-delegation\nidentity swap]:::step
    X --> V[(Vault\nper-tool secret)]:::art
    X --> KC[(Keycloak\nRFC 8693 exchange)]:::art
    X --> D[downstream MCP server\nsees the USER]:::step
....

. *Authenticate the workload, not a secret.* SPIRE issues every agent pod a
  SPIFFE JWT-SVID based on what it _is_ (namespace, service account, node) --
  there is no API key to provision, leak, or rotate.
. *Authorize the specific tool.* Kyverno's ext_authz policy decides per-call
  whether this agent may invoke this tool. Deny is the default.
. *Swap the identity.* The `ext-proc-delegation` service exchanges the
  agent's SVID for the *user's* federated token (Keycloak, RFC 8693 + RFC
  7523), scoped to the downstream audience, and fetches any per-tool secret
  from Vault -- in memory, for one request, never written to the pod.
. *Forward as the user.* The downstream MCP server receives the user's
  identity. Its logs, RBAC, and rate limits all apply to the human the agent
  is working for.

If any step in that chain errors, the request is *denied* -- both filters are
required; there is no degrade-gracefully path on the credential chain.

== When the agent needs more: JIT escalation

A denied operation does not end in a dead 403. The agent files a scoped,
time-boxed access request, which becomes a *Gitea pull request*. A human
merging that PR is the approval -- no parallel approval system, no chatops
bot with admin rights. The merge webhook drives `jit-approver` to mint an
ephemeral Kubernetes identity (ServiceAccount + Role + RoleBinding) through
Vault's Kubernetes secrets engine, with the Vault *lease TTL equal to the
approval window*. When the lease expires, Vault deletes the identity. Nothing
needs to remember to revoke it.

== Why this is supportable

Exactly *one* component in the chain is custom code: `ext-proc-delegation`,
a small Go service implementing Envoy's ext_proc protocol. Everything else is
a supported product or an established open-source project: SPIRE (Red Hat
Zero Trust Workload Identity Manager), Keycloak (RHBK), HashiCorp Vault,
Kyverno, agentgateway (Linux Foundation), Kata Containers, OpenShift GitOps.
The pattern -- gateway-mediated identity exchange -- is not bespoke to one
agent framework; any MCP-speaking agent gets the same guarantees with zero
agent-side changes.
