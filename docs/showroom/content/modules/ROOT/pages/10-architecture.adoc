== Design principles

* *One custom component.* The whole credential path rides on a single small Go binary (`ext-proc-delegation`) -- one thing to audit and maintain. Everything else is vendor-supported or established OSS.
* *Authorize, then mutate -- separately.* A policy gate decides allow/deny; a separate step injects the credential. A bug in the mutation layer can never become an access-granting path.
* *Fail closed.* Both gate steps are required filters. Any error returns a denial, never a pass-through.
* *Structural auto-revoke.* Grants are short-lived capabilities and Vault-leased identities; expiry deletes them. No cron, no reconciler, no human in the revoke path.
* *Human approval is a console, not a side channel.* Approvals happen in the approval console, enforcing approver ≠ requester, with a tamper-evident (WORM) record.

== Identity, two ways

The platform issues the agent two identities so it can use both planes:

* a *workload-shaped* identity for the AuthBridge/Keycloak path, and
* a *per-sandbox* identity that carries the user's consent grant for the real-tool (ext-proc) path.

This is what lets one agent both prove "what it is" and act "on behalf of whom"
without ever holding a stored secret.
