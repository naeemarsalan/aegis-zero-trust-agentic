= UC1 — Delegated MCP Credential
:navtitle: UC1 Credential Delegation

[.lead]
An AI agent running in a Kata-isolated pod calls a downstream MCP server (`pfsense-mcp`) through the agentgateway. At no point does the agent hold a credential for pfSense. The downstream server sees the **user's** identity — not the agent's service account — in every request. The agent receives a response with all credential headers stripped.

This is the **no-credential-passing invariant** demonstrated end to end.

== Step-by-step explanation

=== Step 1–2 — SVID fetch

Before making any MCP call, the agent retrieves a **JWT-SVID** from the SPIRE Workload API via the SPIFFE CSI Driver socket. The audience (`aud`) is set to `mcp-gateway` so the gateway can validate the token without trusting a foreign issuer. The SVID subject is `spiffe://anaeem.na-launch.com/ns/agent-sandbox/sa/<sa>` — the agent's exact Kubernetes identity.

The agent pod has `automountServiceAccountToken: false`. The SVID from SPIRE is the only credential it holds.

=== Step 3 — MCP request to gateway

The agent posts a standard MCP JSON-RPC request (`tools/call`) to `https://mcp-gateway.apps.anaeem.na-launch.com/mcp` with `Authorization: Bearer <JWT-SVID>` plus a user context claim.

=== Step 4 — JWT authentication at the gateway

agentgateway validates the SVID against the SPIRE OIDC JWKS endpoint (`https://spire-oidc.apps.anaeem.na-launch.com`). A valid SVID maps its claims into gateway-internal metadata (`dev.agentgateway.jwt`). An invalid or missing token → 401 before any downstream call.

=== Step 5–6 — Kyverno ext_authz (allow/deny)

The gateway calls the **Kyverno authz server** (`ext_authz`) with the tool name, the agent's SVID claims, and the user context. Kyverno evaluates its `ValidatingPolicy` CEL rules and returns **ALLOW** or **DENY**. This decision is made _before_ any credential is minted:

* **DENY** → the gateway returns 403 to the agent. No Keycloak or Vault call has been made. If the agent has a JIT path configured, this 403 triggers UC2.
* **ALLOW** → the gateway proceeds to the ext_proc filter.

=== Step 7 — ext_proc filter invoked

The gateway calls `ext-proc-delegation` over a gRPC bidirectional stream, forwarding the request headers and buffered body. This is the only custom-code hop in the entire chain.

=== Step 8 — Token exchange (Keycloak)

`ext-proc-delegation` executes a two-leg Keycloak exchange:

* **Leg 0/1 (RFC 7523 jwt-bearer):** presents the agent's SPIFFE JWT-SVID as a JWT client assertion to Keycloak's token endpoint. Keycloak validates the assertion against the SPIRE OIDC JWKS and issues a realm token bound to the user identity. This leg uses a RHBK preview feature; a `mode: legacy` fallback is available (ADR 0003).
* **Leg 2 (RFC 8693 token exchange):** exchanges the realm token for one scoped to the downstream audience (`pfsense-mcp`). The result is a short-lived access token that carries the **user's** identity, scoped to exactly one downstream service.

=== Step 9–11 — Vault secret fetch

`ext-proc-delegation` authenticates to Vault using its own JWT-SVID via the `auth/jwt` mount (which trusts SPIRE OIDC). Vault returns a token scoped to the per-tool secret path. The service reads the per-tool secret (e.g., the pfSense API key) into memory. The secret is never written to disk, never logged, and discarded after the request completes.

=== Step 12 — Header mutation

`ext-proc-delegation` rewrites the outgoing request headers:

* **Sets** `Authorization: Bearer <user-scoped token>` (the RFC 8693 result — user identity, downstream audience)
* **Sets** the per-tool secret header (e.g., `X-PfSense-Token`)
* **Clears** the agent's original SVID from the `Authorization` header

The mutated headers are returned to agentgateway, which forwards the request to `pfsense-mcp`.

=== Step 13–14 — Upstream call with user identity

agentgateway forwards the mutated request to `pfsense-mcp:8000`. The upstream MCP server sees:

* `Authorization: Bearer <user token>` — the authenticated _user_, not the agent service account
* The per-tool secret in the appropriate header
* No trace of the agent's SVID

`pfsense-mcp` validates the user token against Keycloak's JWKS and logs the user subject for every tool invocation.

=== Step 15–16 — Response and credential strip

The response returns through the gateway. `ext-proc-delegation` processes the response leg and **strips all credential and authorization headers** — so even if the upstream were to echo back a credential in a response header, the agent would never see it. Response body processing is SKIP by default.

=== Step 17 — Audit

`ext-proc-delegation` emits an OpenTelemetry span (request → exchange → upstream) and a structured audit event to Loki. Tool arguments are **SHA-256 hashed** before logging — raw argument values are never written to any log sink.

== What the agent sees vs. what the upstream sees

[cols="1,2,2",options="header"]
|===
| | Agent sees | pfsense-mcp sees

| Identity
| Its own JWT-SVID (before the call); response with no auth headers (after)
| User's RFC 8693 token scoped to `pfsense-mcp` audience

| Credential
| None — no pfSense API key ever reaches the agent pod
| pfSense API key (from Vault, injected by ext-proc)

| Arguments
| Its own MCP JSON-RPC body
| Same body (unmodified by ext-proc)
|===

== Verification commands

[source,bash]
----
# 1. Confirm the agent pod has no credential mounts
oc get pod -n agent-sandbox <pod> -o json \
  | jq '.spec.volumes[] | select(.secret != null)'
# Expected: empty — no Secret volumes

# 2. Confirm pfsense-mcp logs show the user subject, not the agent SPIFFE ID
oc logs -n agentic-mcp deploy/pfsense-mcp | grep "sub" | tail -5

# 3. Confirm audit events in Loki have hashed args
curl -s "http://172.16.2.252:3100/loki/api/v1/query" \
  --data-urlencode 'query={app="ext-proc-delegation"} |= "tool_args"' \
  | jq '.data.result[0].values[-1][1]' | grep "sha256:"

# 4. Confirm the agent's response has no Authorization or credential headers
# (run from within the agent pod after a tool call and inspect returned headers)
----

== Maturity notes

* The RFC 7523 jwt-bearer leg (RHBK preview) is the primary risk. If that leg is unavailable on the deployed RHBK version, the `mode: legacy` flag in `ext-proc-delegation` config switches to the fallback exchange path without a code change (ADR 0003).
* agentgateway is alpha software — CRD schema and filter API may change between builds. The deployment pins the chart version.
* The dangerous-tools JIT gate (blocking destructive pfSense operations until UC2 approval) uses a signed session-capability JWT issued by `jit-approver` and verified stateless by Kyverno (ADR 0006).
