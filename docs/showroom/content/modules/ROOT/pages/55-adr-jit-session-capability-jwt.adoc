= ADR 0006 — JIT Session-Capability JWT for the Dangerous-Tools Gate
:navtitle: ADR 0006: JIT Session-Capability JWT

[.lead]
*Status: Accepted*

== Context

=== The dangerous-tools gate (finding C3/H3 PARTIAL)

The security review addendum graded the dangerous-tools gate *PARTIAL*:

* *C3 (PARTIAL):* The original bypass (a non-empty plain `X-JIT-Session` header)
  was eliminated.
  The gate now requires a cryptographically valid `X-JIT-Session-JWT` — RS256-signed
  against the jit-approver JWKS, with `exp`/`aud`/`iss` validated and the requested
  tool inside a `tool_scope` claim.
  But nothing was minting this JWT (no `/jwks` route, no signing code).
  Dangerous tools were permanently un-approvable.

* *H3 (PARTIAL, N2/N3):* Per-session ephemeral Vault roles fix the
  "advertised ≠ enforced" scope problem.
  But:
** N2: `jit-approver.hcl` granted only `read` on `secret/data/jit/*`; jit-approver
   needs `create`/`update` to write the issuance record.
** N3: The ephemeral Vault role `kubernetes/roles/jit-<session>` was never cleaned
   up, causing roles to accumulate.

=== Why Vault Agent Injector cannot deliver dynamic session credentials

The Vault Agent Injector annotation is part of the pod spec, which is fixed at
pod-start time.
The Vault Kubernetes role `kubernetes/roles/jit-<session>` does not exist until
jit-approver creates it after a human approves the request in the Approval Console.
There is no session-specific Vault path to reference in a static pod annotation.
This is a structural chicken-and-egg: the pod that needs the credentials has already
started before the approval creates the Vault role.

== Decision

=== 1. jit-approver mints the session JWT on issuance

When a session transitions to `issued`, jit-approver:

. Calls `vault write kubernetes/roles/jit-<session>` (ephemeral role)
. Calls `vault read kubernetes/creds/jit-<session>` (mints `sa_token`)
. Mints an RS256 session-capability JWT:

[source,json]
----
{
  "jti": "<session_id>",
  "iss": "https://jit-approver.mcp-gateway.svc.cluster.local:8080",
  "aud": ["kyverno-authz"],
  "tool_scope": ["<approved MCP tool name>", "..."],
  "exp": "<issuance + approved_window_seconds>"
}
----

[start=4]
. Writes both credentials to `secret/data/jit/<session>` (KV v2):

[source,json]
----
{
  "sa_token":    "<Vault-minted k8s SA token>",
  "session_jwt": "<signed RS256 JWT>",
  "expires_at":  "<RFC3339>",
  "namespace":   "<approved namespace>",
  "tool_scope":  ["<tool>", "..."]
}
----

=== 2. Credential delivery via /status over SVID-mTLS

`GET /requests/\{id}/status` returns both fields once `state == issued`.
No separate injection mechanism is needed; the channel is the existing SVID-mTLS
session between the agent and jit-approver.

=== 3. Kyverno gate — stateless, fail-closed

The Kyverno ext_authz policy verifies `X-JIT-Session-JWT` by fetching jit-approver's
JWKS (`/jwks`) and running CEL JWT validation.
Missing, invalid, or expired JWT → 403.
No live HTTP callback to jit-approver per request.

=== 4. Session JWT is the agent's own capability — not a downstream credential

The `session_jwt` proves the agent's approval to the gateway layer (Kyverno).
It does not expose a downstream service credential.
The no-credential-passing invariant is intact.

=== 5. Vault policy fix (N2)

`platform/vault/config/jit-approver.hcl` now grants `create`, `update`, `read`, and
`delete` on `secret/data/jit/*` and `secret/metadata/jit/*`.

=== 6. Ephemeral Vault role reaper (N3)

jit-approver's background reaper, on session expiry:

. Calls `vault delete kubernetes/roles/jit-<session>`
. Calls `vault delete secret/data/jit/<session>` (soft delete)
. Calls `vault delete secret/metadata/jit/<session>` (hard delete)

== Consequences

*Positive:*

* *Stateless gate:* Kyverno verifies the session JWT cryptographically with no live
  dependency on jit-approver.
  A jit-approver pod restart does not affect in-flight approved sessions.
* *No injector bootstrapping problem:* dynamic sessions do not require pod restarts
  or static annotations.
* *Invariant intact:* the session JWT is the agent's own scoped capability; no
  downstream service credentials are held by agent pods.
* *Expiry is cryptographic:* a past-`exp` JWT is rejected without any state lookup.

*Trade-offs:*

* Adds signing key management to jit-approver's responsibilities.
  Key rotation requires a JWKS overlap window for in-flight JWTs.
* The jit-approver RS256 signing keypair must be persisted across pod restarts
  (Vault or a Kubernetes Secret).
* The Kyverno `ClusterCleanupPolicy` backstop only deletes K8s objects, not Vault
  roles — if the reaper fails, `kubernetes/roles/jit-<session>` entries accumulate.
  Monitoring on stale entries is recommended.
