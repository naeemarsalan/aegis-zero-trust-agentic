= ADR 0003 — Keycloak Token-Exchange Strategy
:navtitle: ADR 0003: Keycloak Token Exchange

[.lead]
*Status: Accepted*

== Context

UC1 requires the delegation service to turn the agent's SPIFFE JWT-SVID into the
*user's* federated access token scoped to the downstream MCP audience.
Two standards are in play:

* *RFC 7523 (JWT bearer grant)* — exchange an externally-issued JWT (the SPIRE
  JWT-SVID) for a Keycloak token.
  In RHBK this leg rides a *preview feature*.
* *RFC 8693 (OAuth 2.0 Token Exchange)* — exchange an existing token for one scoped
  to a different audience.
  Keycloak's RFC 8693 support is strongest for internal-to-internal exchange;
  external-token and impersonation cases are more constrained and version-dependent.

The delegation chain is: *leg 0/1* RFC 7523 jwt-bearer (SVID → Keycloak token,
preview) then *leg 2* RFC 8693 exchange (→ downstream audience).
Both legs touch features whose stability varies by RHBK version.

== Decision

Implement the two-leg exchange (RFC 7523 jwt-bearer → RFC 8693 audience exchange)
and gate it behind a *service-level `mode` flag*:

[cols="1,3",options="header"]
|===
| Mode | Path

| `standard`
| Full RFC 7523 (preview) + RFC 8693.
Default when the target RHBK version supports the preview feature.

| `legacy`
| Fallback path that avoids the fragile preview leg (pre-provisioned
federated-identity or direct-grant arrangement) for RHBK versions where the preview
feature is unavailable or unstable.
|===

The flag is configuration (`internal/keycloak` + `internal/config`), selected per
environment, so the same binary runs against either RHBK posture without a code change.

== Consequences

* The PoC is *not blocked* by RHBK preview-feature instability — `legacy` is the
  documented escape hatch.
* Acknowledged limits: RFC 8693 in Keycloak is most reliable *internal-to-internal*;
  external-token (RFC 7523) and impersonation cases carry preview/version risk and
  must be re-validated against the actual RHBK version on `anaeem` before the demo.
* `internal/keycloak` must implement both legs behind one interface and select by
  `mode`; tests must cover both paths and the failure-to-fail-closed behavior.
* Audit/OTel must record which `mode` produced each token — observable per request in
  Loki.
* Downstream audience is an *allowlist* mapped per tool — the exchange may only target
  an approved audience.
