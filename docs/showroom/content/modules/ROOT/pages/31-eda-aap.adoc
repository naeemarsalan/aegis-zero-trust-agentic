= EDA / AAP — Self-Healing Remediation Loop
:navtitle: EDA / AAP

[.lead]
Event-Driven Ansible (EDA) with Ansible Automation Platform (AAP) 2.6 provides the automated remediation backbone.
EDA Rulebooks listen to alert events from the observability stack, match conditions (GPU health degradation, policy violations, anomalous agent behavior), and trigger AAP Job Templates that execute remediation playbooks — creating a closed-loop, human-in-the-loop response pipeline.

The key property: **remediation requires human approval before any destructive change executes**.
EDA triggers the same JIT approval gate as UC2 — the human approves in the Approval Console (mint-gate); no direct privileged change is made without a four-eyes review.

== Deployment model

[NOTE]
====
AAP 2.6 **already exists** at `https://aap-aap.apps.hammer.na-launch.com` on the `hammer` cluster.
This is a **config-only** integration — no new AAP operator or controller is deployed.
All work consists of creating Event Streams, Rulebook Activations, Credential objects, and Job Templates in the existing AAP via its API.

There are no Kubernetes manifests on the `anaeem` cluster for AAP itself.
====

== The self-healing loop

[source]
----
Denial metric or alert fires in agentic-observability
    ↓ AlertManager rule matches
AlertManager fires HMAC-signed webhook to EDA Event Stream
    ↓ HMAC authenticated
EDA Rulebook Activation matches the alert
    ↓
EDA triggers AAP Job Template
    ↓
Job Template runs remediation playbook
    ↓
Playbook submits JIT approval request to jit-approver (if elevated access required)
    ↓ human approves in Approval Console (approver ≠ requester enforced)
jit-approver mints short-lived capability JWT
    ↓ OR → direct application via ArgoCD if the change is GitOps-managed
Remediation applied
----

At no point does EDA or the job template make a direct privileged change.
The PR is the boundary.

== Security posture

* **AAP API credentials:** stored in Vault (`kv-v2/aap/controller-token`); injected into the EDA webhook sender component via Vault Agent Injector; no hardcoded credentials in rulebook YAML
* **EDA Rulebook Activations:** use Credentials of type `Red Hat Ansible Automation Platform` pointing at the Controller
* **AlertManager → EDA:** HMAC-authenticated Event Stream; TLS transport
* **Approval Console (mint-gate) is the approval gate** for destructive remediations — same HITL as UC2; approver ≠ requester enforced, WORM audit record written
* **Fail-mode:** if EDA cannot reach AAP Controller, Rulebook Activations enter `failed` state and alert; no silent failure

== Verify

[source,bash]
----
# 1. Confirm AAP Controller is reachable and authenticated
curl -sk https://aap-aap.apps.hammer.na-launch.com/api/v2/ping/ | jq .version

# 2. List active EDA Rulebook Activations
curl -sk -H "Authorization: Bearer <token>" \
  https://aap-aap.apps.hammer.na-launch.com/eda/api/v1/activations/ | jq '.[].name'

# 3. Confirm an AlertManager alert reaches EDA (trigger a test alert)
oc exec -n agentic-observability deploy/alertmanager -- \
  amtool alert add alertname=test severity=warning
----

== Maturity flags

* AAP 2.6 with gateway API is GA; EDA Event Streams (replacing legacy webhook receivers) is GA in AAP 2.6
* The Approval Console (mint-gate) integration with EDA is a custom pattern — no upstream AAP feature
* AAP Controller 4.7.12 — check Red Hat errata for any CVEs before connecting to the platform network
