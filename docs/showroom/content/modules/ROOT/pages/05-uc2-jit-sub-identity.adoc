= UC2 -- Human-approved, just-in-time elevation
:navtitle: UC2 JIT elevation

[.lead]
When an agent is denied a privileged action, it cannot proceed on its own. It
requests a scoped, time-boxed grant; a *human approves it in the console*; and
the approval mints a *short-lived capability* that does the action and then
expires. The approver is never the requester, and the whole exchange is recorded
in a tamper-evident (WORM) audit log.

== The approval loop

[mermaid,target=uc2-sequence,format=svg]
....
sequenceDiagram
  autonumber
  participant Agent as AI agent (SVID only)
  participant Gate as Zero-trust gate
  participant Console as Approval console
  participant Human as Human approver
  participant JIT as JIT mint-gate
  participant Tool as Real tool + audit

  Agent->>Gate: privileged WRITE
  Gate-->>Agent: 403 grant_scope_denied
  Agent->>Console: request {scope, justification}
  Console-->>Human: pending approval — shows scope & TTL
  Human->>Console: APPROVE (approver ≠ requester)
  Console->>JIT: mint capability
  Note over JIT: scope = approved only<br/>TTL = write 5m single-use / exec 30m
  JIT-->>Agent: short-lived scoped capability
  Agent->>Gate: WRITE + capability
  Gate->>Tool: 200 — real change applied
  Note over JIT,Tool: capability auto-expires — every step WORM-audited
  Note over Agent: riskier follow-on = NEW request, never silent escalation
....

== What makes it safe

[cols="1,2",options="header"]
|===
| Property | How

| Agent cannot self-approve
| The console enforces *approver ≠ requester* at the mint layer -- not by convention.

| Grant is least-privilege
| The capability carries *only* the approved scope; nothing wider is reachable.

| Grant cannot be forgotten
| Short-lived by construction (write = 5 min single-use; exec = 30 min); expiry deletes it.

| Every action is attributable
| Who requested, who approved, the exact scope and TTL -- all in a tamper-evident (WORM) record.

| No silent escalation
| A riskier follow-on action requires a *new* request and a *new* approval.
|===

== From exception to policy

When the same agent keeps requesting the same scope, that access is no longer
exceptional. The pattern is a signal to promote it to a *standing, GitOps-managed
policy* (least-privilege, still attributable) -- after which the JIT path stops
firing for it.

See xref:04-uc1-credential-delegation.adoc[UC1] for the delegated-read half of
the model, or xref:11-security-model.adoc[the security model] for trust boundaries.
